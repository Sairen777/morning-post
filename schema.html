Implementation ledger

What changed

The backend moved from a connector prototype into a multi-user persistence + API system.

Foundations Runtime naming was cleaned up: items now carry feedExternalId instead of the overloaded sourceId. Ruleset selection accepts feed kind. PostgreSQL, Drizzle, migrations, the real-Postgres test harness, Hono app scaffolding, centralized errors, validation, config, and credential crypto were added.
Identity and sessions Users, password hashing, login, logout, profile routes, server-side sessions, HttpOnly cookies, dummy hash timing protection, and auth rate limiting were implemented. Auth responses expose only safe public user fields.
Sources and Telegram connection A Source represents one connected account per connector. Credentials are encrypted and nullable so disconnect can remove custody without deleting history. Telegram web QR login stores the completed session into the source, rate-limits sensitive steps, and prompts users to revoke in Telegram Devices on disconnect.
Feeds A Feed represents one subscription inside a source: a Telegram channel, group dialogue, RSS URL, subreddit, etc. Feed discovery, subscribe/revive, soft-delete, custom prompts, source/feed ordering, and cursor fields were added. Feed external ids are unique only inside their source.
Item ingestion Connectors can fetch only selected feed external ids. Item caches normalized content with UNIQUE(feedId, externalId) upserts so edited messages refresh instead of duplicating. Ingestion advances a feed cursor only after item persistence succeeds.
Summaries Summary stores a feed-period result. Prompt composition is explicit: base prompt, user prompt, feed custom prompt, and kind-specific rules. Empty item windows store empty summaries without calling the model. Summary rows snapshot the feed name.
Digests Digest represents the user's morning post for a period. Sections are derived from summaries, not stored. JSON and Markdown views are exposed. Partial feed failures preserve successful sections and mark the digest failed.
Orchestration and scheduling The orchestrator groups feeds by source, reuses one connector per source, skips already-ingested feeds, isolates source/feed failures, summarizes successful feeds, and assembles the digest. Croner runs UTC five-field schedules with in-process overlap protection; PostgreSQL leases coordinate API instances and per-user uniqueness prevents overlapping digest runs.
Security audit Tests cover cross-user access denial, no leaked secrets in API responses or logs, source re-enable safety after disconnect, Telegram login race conditions, and rate limits for auth plus Telegram connect flows.

Layer map

Data flow, end to end

User connects source→ Discover feeds→ Subscribe feeds→ Scheduler chooses period→ Connector fetches selected feed ids→ Items upserted→ Summaries stored→ Digest derived
Important naming decision: a connector result map key is a feed's connector-native external id, not a database Source. It is only unique inside one source. Runtime items use feedExternalId; persistence resolves it with the current Source.id plus Feed.externalId, or directly through Feed.id after subscription.
Runtime boundaries: Hono is served directly by Bun.serve. Bun uses the ordinary process environment and Node-compatible filesystem APIs, so OS account permissions and deployment sandboxing control local access. Substack HTTPS resolves and rejects non-public addresses, then connects directly to the validated IP through Node TLS while preserving hostname verification; no HTTP proxy hop sits between validation and transport.

Database model

Entities and relationships

Six domain entities, plus Session as auth infrastructure.

User

domain owner

Represents a person using the app. Owns sources, feeds through sources, summaries through feeds, and digests.

  • email is unique.
  • passwordHash stores Argon2id output; plaintext is never stored.
  • systemPrompt is the user's editable taste/interests prompt. There are no tag/profile tables in v1.
  • defaultLanguage and defaultModel optionally steer summaries.
  • createdAt and updatedAt are epoch-ms numbers.

Session

infra table

Backs cookie authentication. It is intentionally outside the six-domain-entity model.

  • Stores tokenHash, not the raw cookie token.
  • Cookie is server-side session style, not JWT.
  • expiresAt and lastSeenAt are epoch-ms numbers.

Source

connected account

One connector account owned by one user: for example, one Telegram account.

  • UNIQUE(userId, connectorId): one source per connector per user in v1.
  • credentials is an encrypted blob or null after disconnect.
  • position is the primary digest grouping/order key.
  • enabled can pause a source without disconnecting it.
  • Disconnect removes credential custody and soft-deletes active feeds while keeping history.

Feed

subscription

A single channel, group, dialogue, subreddit, RSS URL, or similar subscription inside a source.

  • UNIQUE(sourceId, externalId): external ids repeat across users and connectors.
  • kind currently supports news and discussion, and drives prompt rules.
  • customPrompt adds feed-specific steering.
  • position orders feeds inside their source.
  • enabled pauses; deletedAt soft-deletes; lastFetchedPeriodEndMs is the ingestion cursor.

Item

content cache

Stores a cached NormalizedItem so periods can be summarized again without re-fetching external content.

  • externalId is the connector-native item id, e.g. a Telegram message id.
  • payload is jsonb and validated as NormalizedItem.
  • UNIQUE(feedId, externalId) makes windows idempotent.
  • Upsert updates payload and fetchedAt, so edited messages refresh.

Summary

feed-period result

Stores one feed's summary for one explicit period range.

  • UNIQUE(feedId, periodStartMs, periodEndMs).
  • points is jsonb and validated as SummaryPoint[].
  • feedNameSnapshot preserves rendering after rename or soft-delete.
  • generatedAt is the only timestamp; no blanket updatedAt.
  • Rerunning a period replaces the period summary row.

Digest

morning post

The user-facing digest for one period. It stores metadata and status; sections are derived.

  • UNIQUE(userId, periodStartMs, periodEndMs).
  • Status is pending, complete, or failed.
  • Sections come from that period's summaries, ordered by source then feed.
  • Partial failures keep successful sections and set status to failed.

JSONB ledger

intentional blobs only

JSON is used only where the value is polymorphic or read whole.

  • Source.credentials: encrypted credential envelope.
  • Item.payload: cached normalized item.
  • Summary.points: rendered summary points.
  • Everything sorted, joined, filtered, or authorized is normalized.

How it works

Core user and system flows

From registration to a rendered digest.

1. Registration and login

Register. The user posts name, email, password, and profile prompt fields. Password is hashed with Argon2id before storage.
Login. The service uses a dummy hash path for unknown emails so wrong-password and unknown-user timing are closer. Both failures return the same message.
Session. A cryptographically random token is set as a cookie; only its hash is stored in Postgres.
Profile. GET /auth/me and PATCH /auth/me expose/update only public user fields.

2. Telegram connection

Start QR login. POST /connectors/telegram/login creates a pending login session and returns data needed by the web UI to show QR/login status.
Poll status. GET /connectors/telegram/login/:id returns pending, 2FA-required, complete, expired, or error state for the authenticated user only.
Submit 2FA if needed. POST /connectors/telegram/login/:id/2fa rate-limits password attempts.
Persist credentials. A completed session is envelope-encrypted and stored in Source.credentials. Plaintext is disposed from the login manager path.
Disconnect. DELETE /sources/:id removes stored credentials and tells Telegram users to revoke the session in Telegram Devices.

3. Feed discovery and subscription

Discover available feeds. The source credential is decrypted for the owner, a connector is created, and listAvailableFeeds() returns source-scoped feed external ids, names, and kinds.
Subscribe. POST /sources/:sourceId/feeds creates or revives a feed. It rejects disconnected sources and keeps uniqueness per source.
Customize. Users can set kind, customPrompt, position, and enabled.
Unsubscribe. DELETE /feeds/:id soft-deletes the feed. Past summaries remain renderable.

4. Scheduled digest run

Pick users. The scheduler lists users and skips a user if another run is already in progress for them.
Coordinate the tick. Croner evaluates UTC five-field schedules with overlap protection in each process; an atomic PostgreSQL lease permits only one instance to execute the leader tick.
Compute period. If the user has a previous digest, start at latestDigest.periodEndMs + 1; otherwise use a one-day cadence fallback.
Group by source. The orchestrator lists enabled sources and enabled feeds, grouped by source so one connector instance can serve multiple feeds.
Ingest feed windows. Each feed fetches lastFetchedPeriodEndMs + 1 through the period end, or a default lookback for new feeds.
Upsert items. Normalized items are validated, checked against the expected feedExternalId, stored, and only then the cursor advances.
Summarize. Each successful feed-period is summarized or retrieved from cache. Empty windows store empty summaries.
Assemble digest. Sections are derived from summaries, grouped by source, ordered by Source.position and Feed.position or name.

Security model

Security decisions and invariants

The highest-risk asset is the Telegram session string.

Credential encryption

Credentials are envelope-encrypted before storage:

  1. Generate a fresh random 256-bit data key per encryption.
  2. Encrypt plaintext with AES-256-GCM and a fresh 96-bit IV.
  3. Bind ciphertext to [userId, connectorId] with AEAD additional authenticated data.
  4. Wrap the data key through a KeyProvider.
  5. Store v, wrappedDataKey, iv, and ciphertext.

Honest residual risk

This is not zero-knowledge. The server must decrypt credentials while the user is offline to build scheduled digests. Encryption protects DB dumps, backups, read-only SQL leaks, and row transplant attacks. It cannot stop a currently rooted app host from using the key in process.

Security controls now in place

RiskCurrent control
Password theft from DBPasswords are Argon2id hashes, never plaintext.
Cookie token replay from DBSessions store tokenHash, not the raw token.
User enumerationLogin failures use the same message for unknown email and wrong password.
Brute force / resource abuseRegister, login, Telegram login start, and Telegram 2FA are rate-limited.
IDOR / cross-user readsRepositories and services scope access through authenticated userId; audit tests cover feeds, digests, Markdown output, and cached summary paths.
Credential leakage through APIPublic source serializers exclude credential blobs; audit tests assert no session string, wrapped data key, ciphertext, password hash, or raw cookie token leaks.
Credential row transplantAES-GCM AAD binds encrypted credentials to owner metadata.
TOCTOU around disconnected sourcesRe-enable and subscribe paths check credential presence atomically / transactionally.
Telegram login racesPending capacity is reserved before async client creation and cleaned on startup failure/expiry.
Production hardening path: replace the env-backed EnvMasterKeyProvider with a KMS/Vault provider. The code already has a KeyProvider seam, so KMS can wrap/unwrap data keys without changing CredentialCipher. This moves the master key out of DB backups and off the VPS disk, gives audit logs, and creates a revocation kill-switch.

Local operation

How to run it locally

Backend-only, with Postgres and env configuration.

Prerequisites

  • Bun 1.3.14
  • Node.js 22.13+ for the frontend and Playwright toolchain
  • PostgreSQL 16+ or Docker Compose
  • OpenSSL to generate a credential master key
  • Telegram API ID/hash from my.telegram.org/apps
  • Gemini/OpenAI-compatible API key if you want real summarization

Environment

Repository scripts explicitly load .env.production.local through Bun. Keep that credential-bearing file local and configure it for the command you run.

DATABASE_URL=postgres://morningpost:morningpost@localhost:5432/morningpost

CREDENTIAL_MASTER_KEY=<openssl rand -base64 32>
TELEGRAM_API_ID=...
TELEGRAM_API_HASH=...
GEMINI_API_KEY=...

What remains additive

Future extensions

These are deliberate deferrals, not broken pieces of the current architecture.

KMS-backed credentials

Replace EnvMasterKeyProvider with a KMS/Vault provider. Add key identifiers for easier rotation, encrypted backups with separate keys, and eventually per-user KMS keys.

More connectors

RSS, Reddit, Substack, YouTube comments, and others can reuse Source, Feed, Item, Summary, and Digest. No per-connector tables are needed unless a connector has truly queryable data that cannot live in meta.

Frontend

The current work is backend-only. A web UI can be layered on top of registration/login, Telegram QR connection, source/feed management, and digest views.

Audit runs

A Run table for job audit history is still deferred. Add it once scheduler history is useful enough to query.

Email and presenters

Digest data is stored and views are derived. Email delivery or other presenters can read the same DigestView and render to a new target.

Personalization beyond prompts

Automatic theme classification, canonical tags, embeddings, and model routing are future enhancements. v1 keeps user taste in User.systemPrompt and per-feed steering in Feed.customPrompt.

Reviewer checklist

What to keep checking

These invariants are more important than individual files.

  • Every data access path is scoped by authenticated userId.
  • No credential plaintext, encrypted blob internals, password hashes, raw session tokens, or Telegram sessions appear in API responses or logs.
  • Feed external ids are never treated as globally unique; they are source-scoped.
  • Disconnect removes credential custody without deleting historical summaries/digests.
  • Summaries snapshot feed names so history survives feed rename/removal.
  • Digest sections remain derived from summaries; no separate section table in v1.
  • All cross-layer timestamps stay epoch milliseconds.
  • JSONB remains limited to encrypted credentials, cached item payloads, and summary points.
  • Prompt text belongs in src/summarizers/prompts.ts, composed by service code, not scattered through routes.
