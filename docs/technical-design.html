Current architecture, security posture, and scalability review

This document captures the repository architecture as implemented on 2026-07-21 and records the security and scalability decisions for the current Bun 1.3.14/Hono API, SolidStart web UI, PostgreSQL persistence layer, Telegram and Substack connectors, and OpenAI-compatible summarization pipeline.

Executive summary

Architecture is cleanly layered; production hardening is not complete.

What is strong
  • Repository functions consistently accept userId and scope reads/writes to ownership.
  • Session tokens are random 256-bit values; only SHA-256 token hashes are stored.
  • Connector credentials use envelope encryption with AES-256-GCM and owner-bound AAD.
  • Item and summary writes are idempotent through database unique constraints and upserts.
  • Service seams are simple: connectors fetch/normalize, summarizers summarize, repositories persist.
What blocks safe production scaling
  • Bun processes inherit the operating-system privileges of their account; filesystem, environment, network, subprocess, and native-module boundaries must therefore be enforced by deployment isolation and application code.
  • Rate limiting is process-local and trusts request forwarding headers without an app-level trusted-proxy policy.
  • Croner prevents overlapping callbacks within a process, while PostgreSQL scheduler leases coordinate digest ticks across API instances.
  • Summarization is a provider boundary with deployment-wide text and vision endpoints, deterministic text/item/image caps, and run-local vision fallback behavior.
  • Telegram/media artifacts are written locally with no TTL, quota, or object-storage lifecycle.
Important corrections
  • No current digest-run IDOR was confirmed: GET /digests/runs/:id first loads the run with findDigestRunForUser(database, id, context.var.userId), then lists feed rows for the owned run.
  • listDigestRunFeedsForRun(database, runId) is still a fragile repository helper because it is not self-scoped; pass userId if another caller is added.
  • upsertDigestForPeriod is already atomic: it uses insert(...).onConflictDoUpdate(...).returning(). There is no current check-then-insert race in digest upsert.
Security decision: Morning Post is not zero-knowledge. Scheduled digests require the backend to decrypt connector credentials without the user present. The honest target is narrower: a database or backup leak alone should not expose credentials; provider boundaries should be explicit; operational access should be minimized and auditable.
System context

Morning Post is a multi-user digest service with external connector and LLM boundaries.

The web UI calls the Bun API with cookie credentials. The API owns authentication, connector credential custody, feed subscription state, ingestion, summarization, digest assembly, and scheduled runs. PostgreSQL stores users, sessions, encrypted source credentials, feed subscriptions, item cache, summaries, digests, and digest-run audit rows.

Trust boundaries

  • Browser → API: cookie-authenticated requests. Origin guard middleware on unsafe requests validates Origin or Referer against the configured ALLOWED_ORIGINS. Security headers include HSTS, frame denial (DENY), and nosniff. Body size limited by MAX_REQUEST_BODY_BYTES.
  • API → PostgreSQL: persistence for sensitive metadata and encrypted credentials. DB_SSL_MODE configures TLS (default disable for local loopback; require or verify-full for remote). Pool size, idle timeout, and connect timeout are explicitly configured from env vars.
  • API → Telegram: Telegram session credentials are full-account bearer tokens. Server custody is unavoidable for scheduled digests.
  • API → Substack: users explicitly import substack.sid and may also provide connect.sid when their browser has it; Morning Post never accepts a Substack password or raw cookie header. Account-subscription discovery and paid/full post retrieval send only the provided session cookies to the fixed https://substack.com origin. Manual publication validation fetches public archives without cookies through the IP-pinned HTTPS boundary. The integration replays an unsupported browser session and may conflict with Substack terms; the UI discloses that risk before connection.
  • API → filesystem: media are local files with feed-isolated paths, per-connector quota (MEDIA_QUOTA_BYTES), per-period cleanup, and weekly TTL sweep (MEDIA_TTL_MS). Bun's Node-compatible filesystem APIs operate with the process account's privileges rather than runtime permission flags. Deployment ownership and filesystem ACLs should restrict the process to every connector-specific media directory from CONNECTORS_MEDIA_DIR (telegram_media, substack_media, youtube_media, reddit_media, x_media, rss_media) plus the shared/legacy ./media; application path validation, feed isolation, quotas, eviction, cleanup, and TTL traversal remain the in-process boundaries.

Runtime and startup

Local startup pins the Bun toolchain, listener addresses, scheduler semantics, and browser proxy path.

The backend runtime and workspace package manager require Bun 1.3.14; PostgreSQL and OpenSSL are service/cryptography prerequisites. Node.js 22.13+ remains a frontend and Playwright toolchain prerequisite because apps/web declares that engine; it is not the top-level package manager. bun install installs the root package and the apps/web workspace. Root backend scripts explicitly load .env.production.local, so startup does not depend on implicit environment-file discovery. Canonical operations are bun run db:migrate, bun run dev:api, bun run start, bun run dev:cli, bun run test, bun run typecheck, bun run web:test, and bun run web:e2e; generation, cleanup, fixture capture, web development, build, preview, and focused typecheck commands use the other exact scripts declared in package.json.

Current runtime: Hono is served by Bun.serve. Croner 10.0.1 schedules retained jobs in UTC with five-part expressions, rejects duplicate job names, uses protect: true to skip overlapping callbacks within a process, sanitizes and logs callback errors through its catch handler, and stops and clears every retained job when CronScheduler.stop() is called. Digest ticks additionally acquire the existing PostgreSQL scheduler_leases lease so only one API instance performs the work.

Domain model

The persistence model is generic across connectors.

The app models connector accounts as sources and individual subscriptions as feeds. Items and summaries are cached per feed. Digests are user-facing period aggregates derived from summaries. Digest-run tables are operational audit trails for each manual or scheduled run.

One running digest per user is intentional. After a Dashboard reload, the initial run-status check restores any persisted running state and duplicate submission remains blocked until that check settles. While a run is active, the Dashboard polls its status every five seconds without overlapping requests and refreshes the digest list when the run reaches a terminal state. If POST /digests/run loses a race and returns 409, the Dashboard maps the conflict to the active-run UI instead of displaying raw error text.

API startup and every scheduler leader tick call recoverStaleDigestRuns(). The recovery transaction marks only runs older than DIGEST_RUN_STALE_AFTER_MS and their still-running feed stages as failed; terminal feed stages remain unchanged. Fresh running rows are preserved so a rolling multi-instance deployment cannot fail work owned by another live process.

Substack archive ingestion accepts both newsletter articles and podcast episodes as summarizable posts; unrelated archive entry types remain excluded. Paid access is an explicit entitlement boundary, not a body-shape heuristic. An authenticated GET /api/v1/posts/by-id/:id can return a nonempty teaser in post.body_html to a free subscriber, so neither body presence nor text length proves access. SubstackSessionClient reads subscription.membership_state; only the exact value paid_subscriber is affirmative. Missing or unknown evidence is normalized conservatively to meta.hasPaidSubscription: false. For an only_paid archive item without entitlement, normalization falls back to the preview and emits meta.contentAccess: "preview" plus meta.hasPaidSubscription: false. Entitled paid responses must contain a nonempty normalized private body; otherwise ingestion fails rather than summarizing or caching the public teaser. Inaccessible paid posts persist as contentAccess: "paid" with no points and never trigger a model call.

Every digest rerun authoritatively re-ingests the full requested period for each feed containing an only_paid Substack Item, even when its cursor already covers that period. This symmetric refresh observes free-to-paid upgrades and paid-to-free downgrades. Historical refreshes preserve the maximum cursor endpoint instead of moving it backward. The same path repairs legacy Items whose entitlement flag is absent without guessing or a destructive migration. Item fetchedAt advances only when the payload changes, making it a durable content-version timestamp. Any changed or newly inserted Item newer than its cached Summary forces regeneration, including public podcasts first discovered during paid-feed repair; if the model call fails, persisted timestamps remain stale and the next run retries.

Every DigestView includes paidPosts. Digest projection partitions paid articles out of ordinary sections, preserving stable digest/article order for title projection only when the owning Substack source has showPaidPostTitles enabled. An Item's explicit meta.hasPaidSubscription === false overrides stale cached summary points during projection; a missing flag is never guessed to mean either paid access or inaccessibility. The API can retain a genuinely empty article section, but the web digest omits empty Substack publication sections instead of rendering No articles available.. Within a rendered Substack article, the linked title is the source, so article summary bullets do not repeat point-level source links; aggregate connector bullets retain source links. The Markdown API representation and web digest append a final Paid posts section. Paid posts are grouped by newsletter: within the section, each unique newsletter name (from summary.feedNameSnapshot captured at generation time) appears once as a newsletter header, followed by that newsletter's title-only article bullets in their original digest/article order. Only safe HTTP(S) titles are linked. Paid article preview/body text is never rendered there.

Current failure isolation

A connector creation failure records a failed connector-stage run feed. Per-feed ingestion and summarization failures are caught and recorded; successful feeds still contribute to a partial digest. If an unexpected error escapes those boundaries after the run row exists, runForUser() makes a best-effort transition of that exact run to failed with sanitized error text, then preserves and rethrows the original error.

Security review

Primary risks are permission breadth, provider boundaries, and edge hardening.

Security controls already in place

Control Observed implementation Why it matters
Password hashing hash-wasm Argon2id with 19 MiB memory, 2 iterations, parallelism 1. Baseline resistance to offline password cracking; parameters are encoded in the hash.
Login enumeration resistance Unknown email and bad password return the same message; dummy hash path is verified. Reduces user enumeration and simple timing leaks.
Session token storage Raw 256-bit token goes only to cookie; database stores SHA-256 hash. A session table leak does not directly yield replayable cookies.
Cookie flags httpOnly: true, secure: true, sameSite: "Lax", path: "/". Protects the token from JavaScript read access and normal cross-site form posts.
Credential encryption Per-call random data key, AES-256-GCM, owner AAD, wrapped data key, KeyProvider seam. A database or backup leak alone should not expose connector credentials.
Telegram client identity TelegramClient receives nonempty deviceModel and systemVersion strings instead of deriving and disclosing host operating-system details through node:os.type() and node:os.release(). Supplying stable client identity strings minimizes host fingerprinting. Bun's Node-compatible system APIs use the process account's privileges, so deployment isolation—not runtime flags—controls access to host information.
Substack credential minimization Strict schemas require substack.sid and accept connect.sid when the authenticated browser provides it. Session validation reads the user ID from userSettings[0].user_id (also accepting the documented user_settings alias), distinguishes provider/protocol failures from an expired-session 401, and revalidates before fetching subscribed publications from the fixed Substack origin. Subscription discovery joins publication records to subscription publication IDs, paginates by provider cursor, preserves provider order, and rejects repeated cursors. The web client clears local secret state after submission; API responses expose only public source or publication-feed metadata. Avoids password custody, raw-cookie ingestion, accidental credential echoing, cross-origin cookie forwarding, and rejecting valid accounts that do not receive the compatibility cookie.
Substack SSRF boundary Publication URLs require HTTPS domains without credentials, ports, or IP literals. A and AAAA answers must all be global unicast. Each request and redirect connects to the validated IP literal, then uses the original hostname for TLS SNI and certificate verification. Subscribed-publication domains are discovery candidates only; selecting one still runs the same archive validation before persistence. Closes DNS-rebinding TOCTOU while supporting arbitrary custom publication domains, including domains returned by account discovery, under unrestricted API network permission.
Ownership scoping Repositories generally take userId and include ownership filters; route handlers use requireAuth. Prevents cross-user object access at the persistence layer.
Input validation Zod schemas validate bodies, params, row projections, JSON columns, and tagged summary content. Protects service logic from malformed inputs and schema drift.
Error responses Known AppError types return structured errors; unknown failures return generic 500s. Reduces accidental stack/SQL leakage through HTTP.
Prompt locality System prompts live in src/summarizers/prompts.ts. Keeps provider instructions reviewable instead of scattering prompt text across services.

For POST/PATCH/DELETE, validate Origin or Referer against configured origin. SameSite=Lax and JSON preflight are helpful but should not be the only CSRF defense for cookie auth. Remediated. secureHeaders() applied globally: X-Frame-Options: DENY, HSTS max-age=31536000; includeSubDomains, X-Content-Type-Options: nosniff. Origin guard middleware on unsafe requests validates Origin or Referer against ALLOWED_ORIGINS. Body limit (MAX_REQUEST_BODY_BYTES) enforces 413 for oversize payloads. See src/server/app.ts, src/server/middleware/origin-guard.ts. Tests in tests/server/security-audit.test.ts.
S4 Rate limiting is process-local and can be bypassed in multi-instance deployments. rate-limit.ts uses a module-level Map. The default key trusts x-forwarded-for or x-real-ip before falling back to global. For one VPS, document as best-effort only. Before public multi-instance deployment, use Redis/Postgres-backed shared counters and accept forwarding headers only from a trusted proxy path.
S5 Database transport security is not enforced by code. src/db/client.ts calls postgres(url) directly. README examples use local postgres://... URLs. For remote production DBs, require sslmode=require or verify-full in DATABASE_URL, or enforce TLS in client configuration. Local Docker DB can stay plaintext on loopback.
S6 Session hardening is good but not complete for a credential-holding service. Cookie name is session, TTL is 30 days, touchSession() exists in the repository layer but requireAuth currently only validates and sets userId. Rename cookie to __Host-session. Decide shorter absolute TTL and sliding last-seen behavior. Add MFA/WebAuthn/TOTP to the security roadmap before broader multi-user exposure. Remediated. Cookie renamed to __Host-session. Stable token (concurrent requests stay valid). 30-day idle expiry; touchSessionIfDue() extends without changing token; requireAuth sets refreshed Set-Cookie when refreshExpiresAt is non-null. MFA/WebAuthn/TOTP deferred. See src/auth/session-service.ts, src/repositories/session-repository.ts. Tests in tests/server/session.test.ts, tests/server/security-audit.test.ts.
S7 Operational error messages are sanitized but not key-redacted. summarizeErrorForOps() strips Drizzle params and truncates at 500 chars, but does not redact provider keys or credential-bearing URLs. Add redaction patterns for known API-key shapes and URL userinfo before persisting digest_run_feeds.errorMessage. Remediated. Shared secret-redaction utility replaces API-key forms (sk-..., AIza..., xai-..., gsk_..., Bearer ...), PEM private-key blocks, and URL userinfo with [REDACTED] before logging or persistence. Preserves existing Drizzle params stripping and 500-char cap. See src/server/errors.ts, src/server/error-sanitizer.ts. Tests in tests/server/errors.test.ts.
S8 No request body size limit was observed at the Hono app boundary. Routes call context.req.json() directly; no body-size middleware appears in buildApp(). Add a conservative JSON body limit. Most API bodies should be small; connector auth and profile updates do not need large payloads. Remediated. bodyLimit({ maxSize: options.maxRequestBodyBytes }) before routes in buildApp(). PayloadTooLargeError returns 413 PAYLOAD_TOO_LARGE. See src/server/app.ts, src/server/errors.ts. Tests prove oversized JSON returns 413 in tests/server/security-audit.test.ts.
S9 listDigestRunFeedsForRun is a fragile helper, not a current exposed IDOR. The current route first scopes findDigestRunForUser(database, id, context.var.userId) and only calls the helper with the owned run.id. The helper itself filters only by runId. Pass userId into the helper and join/filter through digest_runs.userId so future callers cannot accidentally expose arbitrary run feed rows. Remediated (defense-in-depth). listDigestRunFeedsForRun now accepts userId and joins through digest_runs.userId. The original route was already user-scoped before calling the helper, so no confirmed IDOR existed. See src/repositories/digest-run-repository.ts.
Highest-impact deployment control: run Bun under a dedicated, minimally privileged operating-system identity and enforce filesystem, secret, subprocess, and network restrictions outside the runtime. This reduces the blast radius of a compromised dependency, connector bug, native module, or unexpected code-execution path while application-level validation protects each exposed boundary.
Scalability review

The current shape supports one process and moderate usage; the next bottlenecks are connector batching, LLM payload limits, and distributed coordination.

Scalability strengths

Strength Observed implementation Why it scales
User paging Scheduler pages users by (createdAt, id) with default page size 100. A daily tick does not need to load all users into memory at once.
Bounded per-process user concurrency DEFAULT_USER_CONCURRENCY = 5 in digest-job.ts. Prevents one process from starting all users at once.
Idempotent item cache items_feed_id_external_id_unique and upsert behavior. Overlapping windows can refresh changed items without duplicate rows.
Idempotent summaries and digests Summary and digest repositories use onConflictDoUpdate. Retries can converge to one row per feed-period or user-period.
Partial failure model Digest-run feed rows track connector, ingestion, and summarization stages. Operators can see which feed failed without throwing away successful feeds.
Historical digest resilience Feed soft delete and summary feed-name snapshots preserve old digests. User history survives feed removal and upstream rename.

Scalability findings

Priority Finding Evidence Recommended design Status
P1 Ingestion repeats connector work once per feed within the same source. ingestUserFeeds() groups feeds by source and creates one connector handle, but then loops feeds and calls ingestFeed(). ingestFeed() calls connector.getNormalizedData(window.from, window.to, [feed.externalId]). Add ingestFeedsForSource(): compute one source window, call getNormalizedData(from, to, externalIds) once, validate items per feed, then upsert each feed in its own transaction or bounded batch. Remediated. ingestFeedsForSource() computes union window, calls getNormalizedData once per source, filters returned items per feed, validates and upserts each feed independently. Connector-level exception marks all pending feeds failed; per-feed errors isolated. Single-feed path kept. See src/services/ingestion-service.ts, src/services/orchestrator.ts. Tests in tests/services/ingestion-service.test.ts, tests/services/orchestrator.test.ts.
P2 Summarization has no hard context, item, image, or byte cap. buildContentParts() contains a warning that 300+ group messages may overflow context, but still sends all selected messages/media in a single provider request. Introduce a summarization budget: text token/byte cap, image count cap, chunked map-reduce summarization for long threads, and explicit “truncated due to budget” metadata. Remediated. Aggregate mode uses deterministic sequential chunking with maxTextBytesPerChunk (120k), maxItemsPerChunk (50), and maxImageBytes (1M). Large chunk-result sets are reduced through hierarchical text-only merge batches bounded by serialized UTF-8 content bytes and an item cap of max(2, SUMMARIZER_MAX_ITEMS_PER_CHUNK). An odd level may have a trailing singleton; the guaranteed invariant is that every non-final level reduces item count, not that every non-final batch has fan-in two or that a configured item cap of one is strictly honored. The hierarchy ends with one final merge. Article mode accepts one item, splits oversized text at UTF-8 code-point boundaries, sends media only with the first chunk, and concatenates chunk points without a cross-article merge. SUMMARIZER_TIMEOUT_MS is the per-attempt model request timeout. Transient HTTP fetch/body TypeError, HTTP 429/503 responses, and internal per-attempt TimeoutError retry up to three total attempts; caller cancellation and nonretryable errors do not. A distinct whole-operation watchdog conservatively covers all attempts, distinct vision analysis, and hierarchical merges, capped at three hours. Distinct vision responses trim, exact-deduplicate, and merge duplicate expected item indexes in provider order; missing or invalid indexes retain the nonterminal text-only fallback.
P3 Scheduler coordination is process-local. scheduleDigestJob() registers a scheduler callback in each process. runningUsers is a module-level Set and only prevents overlap inside one runtime. Before running multiple API instances or workers, guard scheduled work with Postgres advisory locks or a dedicated job table with leased rows. Remediated. CronScheduler retains uniquely named Croner jobs configured with timezone: "UTC", mode: "5-part", protect: true, and a sanitizing catch handler; stop() stops and clears all retained jobs. The scheduler_leases table with primary key name provides cross-process coordination: tryAcquireSchedulerLease("digest-job") runs before every tick and only the winner calls runDigestTick. See src/scheduler/digest-job.ts, src/scheduler/scheduler.ts, src/server/main.ts. Tests in tests/scheduler/digest-job.test.ts and tests/scheduler/scheduler.test.ts.
P4 In-flight digest runs have no crash recovery state transition. runForUser() creates digest_runs.status = running and marks finished at the end. A process crash between those points leaves running rows. On startup or scheduler tick, mark stale running runs older than a threshold as failed/stale, then retry based on period and idempotent upserts. Remediated. runForUser() immediately finalizes unexpected in-process failures as failed without masking the original error. For process death or a database outage during finalization, API startup and each scheduler leader tick call recoverStaleDigestRuns(). It transactionally updates running rows older than DIGEST_RUN_STALE_AFTER_MS (default 15 min) and their still-running feed stages to failed with redacted error text. Fresh rows remain untouched for multi-instance safety. The partial unique index on digest_runs(userId) WHERE status='running' prevents duplicate active work. See src/server/main.ts, src/services/orchestrator.ts, and src/repositories/digest-run-repository.ts. Tests in tests/server/main.test.ts, tests/services/orchestrator.test.ts, and tests/db/digest-run-repository.test.ts.
P5 Local media retention is unbounded. Connector media paths use local media directories; README notes Telegram photos are downloaded for channels. No TTL, quota, or cleanup worker is present in the reviewed flow. Add lifecycle: content-addressed paths or per-run dirs, reference counting or age-based cleanup, disk quota alerts, and eventual object storage if multi-host. Remediated. Feed-isolated paths (telegram_media/<feed-key>/<message-id>.jpg). Per-connector quota (MEDIA_QUOTA_BYTES, default 500MiB) evicts oldest before writes. Best-effort per-period cleanup after successful summarization. Weekly TTL sweep deletes files older than MEDIA_TTL_MS (default 7 days). Object storage deferred. See src/connectors/telegram/telegram-connector.ts, src/services/summarization-service.ts. Tests cover quota eviction, TTL cleanup, success/failure cases.
P6 No timeout/cancellation boundary around external connector and summarizer calls. fetch() in the summarizer has retries for 429/503 but no AbortSignal. Connector calls are awaited without a visible timeout wrapper. Add deadline propagation from digest run to connector and summarizer calls. Record timeout as a first-class operational error. Remediated. Optional AbortSignal added to Connector.getRawData, getNormalizedData, getMessagesFromEntity. ingestFeedsForSource creates a per-source timeout from CONNECTOR_TIMEOUT_MS (default 120s) and races it against the connector call via Promise.race. ingestFeedsIndividually uses Promise.race against a Promise.withResolvers rejection. Neither path relies on the connector honoring AbortSignal: late returns after the deadline produce per-feed connector deadline exceeded errors without persisting data or advancing cursors. Telegram connector checks the signal around network-heavy operations. Summarizer calls use SUMMARIZER_TIMEOUT_MS as a per-attempt request deadline: transient fetch/body TypeError, 429/503, and internal per-attempt TimeoutError receive at most three total attempts with fresh deadlines, while caller cancellation and nonretryable errors stop immediately. A separate operation watchdog envelops retry allowance, distinct vision requests, and hierarchical merging, conservatively capped at three hours. Recovered attempts are intentionally absent from .debug_logs/operations.jsonl; only terminal chunk/merge/feed failures and vision fallback are operational events.
P7 Summaries are generated sequentially per feed. assembleDigestForPeriod() loops feeds and awaits summarizeOwnedFeedPeriod() one at a time. Add small configurable summarizer concurrency after context caps are in place. Keep per-provider rate limits and per-user fairness. Remediated. SUMMARIZATION_CONCURRENCY (default 2) with worker-pool helper. Shared OpenAICompatibleSummarizerService handles per-request retries; no unbounded Promise.all. See src/services/digest-service.ts. Tests in tests/services/digest-service.test.ts.
P8 Digest listing is not explicitly paginated at the route/API shape. GET /digests returns listDigestViewsForUser(database, userId). Digest-run listing has a default limit in the repository; digest listing should expose pagination consistently. Add cursor or limit/offset pagination to digest and run listings before histories become large. Remediated. Cursor-based pagination on GET /digests and GET /digests/runs. Opaque base64url cursors with decoded schema validation. Default limit 20, max 100. Response shape: { data, nextCursor }. Web UI "Load more" appends without duplicates, disables while loading, resets after new run/delete/refresh. See src/server/routes/digests.ts, apps/web/src/api/client.ts, apps/web/src/app/Dashboard.tsx. Tests in tests/server/digests.test.ts, frontend unit tests.
P9 Database pool behavior is implicit. src/db/client.ts calls postgres(url) with defaults. Set explicit max connections, idle timeout, connect timeout, and SSL mode per deployment. This matters once scheduler workers and API requests share the same database. Remediated. postgres(url, { max, idle_timeout, connect_timeout, ssl }) with values from getConfig(). DB_SSL_MODE accepts disable (local default), require, verify-full; invalid mode rejected at startup. Same options in src/db/testing.ts. See src/db/client.ts, src/db/testing.ts. Config tests prove remote settings require TLS.
P10 Digest upsert is already race-safe. upsertDigestForPeriod() uses insert(...).onConflictDoUpdate(...).returning() on (userId, periodStartMs, periodEndMs). No change needed for a check-then-insert race. Keep this pattern for future period-scoped tables. No change needed. Atomic INSERT ... ON CONFLICT DO UPDATE already prevents a check-then-insert race. Documented to preserve the pattern for future period-scoped tables.
Prioritized roadmap

Do hardening before horizontal scale; do payload budgets before summarizer concurrency.

Phase 1 — production safety baseline

Work Why first Acceptance criteria Status
Constrain Bun production process privileges. Reduces blast radius across every component, including Node-compatible native, filesystem, DNS, and TLS boundaries. Production runs under a dedicated process identity with deployment-level filesystem, environment-secret, subprocess, and network restrictions; application controls continue to validate connector destinations, pinned HTTPS, media paths, database ownership, and request rates. Deployment responsibility. Bun has no named runtime permission profiles. Repository application boundaries are implemented; operators must apply operating-system, container, or service-manager isolation appropriate to the deployment.
Add HTTP hardening middleware. Cookie-authenticated app needs explicit browser security posture. Responses include HSTS, nosniff, frame denial, referrer policy, and a documented CSP strategy. State-changing routes reject unexpected Origin/Referer. Completed. secureHeaders(), origin guard, bodyLimit().
Add request size limits and operational error redaction. Small controls prevent cheap memory and leakage failures. Oversized JSON request returns a controlled 413. Provider/API-key patterns and credential-bearing URLs are redacted before persistence. Completed. MAX_REQUEST_BODY_BYTES; shared secret-redaction utility.
Define LLM privacy policy in configuration. User messages and media can leave the app. Admin/user-visible setting states hosted vs local provider. The app can refuse hosted LLM calls for protected feeds or deployments. Completed. ALLOW_REMOTE_SUMMARIZATION (default false); loopback always allowed.
Enforce summarizer budgets. Prevents context overflow, unbounded cost, and giant media payloads. Long threads are chunked or truncated deterministically with recorded metadata. Tests cover text-only, media-heavy, and empty-after-filter cases. Completed. Sequential chunking with text/image/item caps; bounded merge requests.

Phase 2 — multi-user and multi-instance readiness

Work Why second Acceptance criteria Status
Shared rate limiter. Required once more than one API process can receive traffic. Postgres counters enforce the same limits across instances and ignore spoofed forwarding headers unless request came from a trusted proxy. Completed. rate_limit_buckets table; trusted proxy via TRUSTED_PROXY_COUNT.
Distributed scheduler lock or leased job table. Prevents duplicate scheduled digests in multi-instance deployment. Only one worker processes a user-period at a time. Stale leases expire safely. Completed. scheduler_leases table with atomic tryAcquireSchedulerLease.
Crash recovery for digest runs. Operational truth must survive process death. Startup/tick reconciles stale running rows to failed and can retry idempotently. Completed. recoverStaleDigestRuns() runs at API startup and leader tick start, transactionally failing stale parent runs and running feed stages while preserving fresh multi-instance work. Partial unique index prevents duplicates.
Explicit database pool and TLS configuration. A scheduler plus web traffic can exhaust implicit defaults. Pool size, connect timeout, idle timeout, and SSL mode are documented and enforced per environment. Completed. DB_POOL_MAX, DB_IDLE_TIMEOUT_SECONDS, DB_CONNECT_TIMEOUT_SECONDS, DB_SSL_MODE.
Credential custody hardening. Telegram session strings are full-account bearer tokens. Master key moves off-box to KMS/Vault/secret manager, backup encryption uses a separate key, and MFA is available for user accounts. Deferred. KeyProvider seam preserved; KMS/MFA/object-storage belong to separate provider/product plans.

Phase 3 — throughput and cost efficiency

Work Why later Acceptance criteria Status
Batch ingestion per source. Removes repeated connector calls and upstream scans. One connector call fetches all feed external IDs for a source-period; per-feed errors remain isolated. Completed. ingestFeedsForSource() with union window and per-feed isolation.
Connector feed lookup caching/direct lookup. Telegram dialog scans are expensive at scale. Requested feeds resolve directly or through cached mapping instead of scanning all dialogs for every run. Not yet addressed. Depends on scheduled-run persistence integration.
Bounded summarizer concurrency. Improves latency after budgets and provider rate controls exist. Configurable per-process concurrency with provider 429 backoff and fairness. Completed. SUMMARIZATION_CONCURRENCY (default 2); worker-pool helper.
Media lifecycle. Needed for disk-bound deployments. Media has TTL/quota cleanup and deterministic paths. Completed. Feed-isolated paths, per-connector quota, best-effort per-period cleanup, weekly TTL sweep. Object storage deferred.
Paginated digest history. Prevents unbounded dashboard payloads. Digest and digest-run APIs expose cursor pagination and the web UI loads incrementally. Completed. { data, nextCursor } response shape; "Load more" in web UI.
