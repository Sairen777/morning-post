Morning Post · Technical Design Documentation

Current architecture, security posture, and scalability review

This document captures the repository architecture as implemented on 2026-07-09 and records the security and scalability review decisions for the current Deno/Hono API, SolidStart web UI, PostgreSQL persistence layer, Telegram connector, and OpenAI-compatible summarization pipeline.

Executive summary

Architecture is cleanly layered; production hardening is not complete.

What is strong
  • Repository functions consistently accept userId and scope reads/writes to ownership.
  • Session tokens are random 256-bit values; only SHA-256 token hashes are stored.
  • Connector credentials use envelope encryption with AES-256-GCM and owner-bound AAD.
  • Item and summary writes are idempotent through database unique constraints and upserts.
  • Service seams are simple: connectors fetch/normalize, summarizers summarize, repositories persist.
What blocks safe production scaling
  • Deno tasks grant broad network, filesystem, environment, system, and FFI permissions.
  • Rate limiting is process-local and trusts request forwarding headers without an app-level trusted-proxy policy.
  • Digest scheduling is single-process: Deno.cron plus an in-memory runningUsers set.
  • Summarization sends all selected items/media in one request with no hard context or payload cap.
  • Telegram/media artifacts are written locally with no TTL, quota, or object-storage lifecycle.
Important corrections
  • No current digest-run IDOR was confirmed: GET /digests/runs/:id first loads the run with findDigestRunForUser(database, id, context.var.userId), then lists feed rows for the owned run.
  • listDigestRunFeedsForRun(database, runId) is still a fragile repository helper because it is not self-scoped; pass userId if another caller is added.
  • upsertDigestForPeriod is already atomic: it uses insert(...).onConflictDoUpdate(...).returning(). There is no current check-then-insert race in digest upsert.
Security decision: Morning Post is not zero-knowledge. Scheduled digests require the backend to decrypt connector credentials without the user present. The honest target is narrower: a database or backup leak alone should not expose credentials; provider boundaries should be explicit; operational access should be minimized and auditable.
System context

Morning Post is a multi-user digest service with external connector and LLM boundaries.

The web UI calls the Deno API with cookie credentials. The API owns authentication, connector credential custody, feed subscription state, ingestion, summarization, digest assembly, and scheduled runs. PostgreSQL stores users, sessions, encrypted source credentials, feed subscriptions, item cache, summaries, digests, and digest-run audit rows.

Trust boundaries

  • Browser → API: cookie-authenticated requests. State-changing endpoints currently rely on SameSite=Lax, JSON content type, and same-origin deployment assumptions rather than explicit CSRF token or Origin enforcement.
  • API → PostgreSQL: persistence for sensitive metadata and encrypted credentials. TLS mode is not enforced in src/db/client.ts; production must enforce it in the connection string or client config when DB is remote.
  • API → Telegram: Telegram session credentials are full-account bearer tokens. Server custody is unavoidable for scheduled digests.
  • API → summarizer provider: message text, prompt text, and selected media leave the app to the configured OpenAI-compatible endpoint unless LOCAL_API=true points to a local provider.
  • API → filesystem: media, eval captures, and result artifacts are local files. They are gitignored, but local retention still needs lifecycle management.

Domain model

The persistence model is generic across connectors.

The app models connector accounts as sources and individual subscriptions as feeds. Items and summaries are cached per feed. Digests are user-facing period aggregates derived from summaries. Digest-run tables are operational audit trails for each manual or scheduled run.

Current failure isolation

A connector creation failure records a failed connector-stage run feed. Per-feed ingestion and summarization failures are caught and recorded; successful feeds still contribute to a partial digest.

Current scalability constraint

Inside a source, ingestion calls connector.getNormalizedData(from, to, [feed.externalId]) once per feed. Connectors already accept multiple external IDs, so the current orchestration repeats upstream work when many feeds share one source.

Security review

Primary risks are permission breadth, provider boundaries, and edge hardening.

Security controls already in place

ControlObserved implementationWhy it matters
Password hashinghash-wasm Argon2id with 19 MiB memory, 2 iterations, parallelism 1.Baseline resistance to offline password cracking; parameters are encoded in the hash.
Login enumeration resistanceUnknown email and bad password return the same message; dummy hash path is verified.Reduces user enumeration and simple timing leaks.

Cookie flagshttpOnly: true, secure: true, sameSite: "Lax", path: "/".Protects the token from JavaScript read access and normal cross-site form posts.
Credential encryptionPer-call random data key, AES-256-GCM, owner AAD, wrapped data key, KeyProvider seam.A database or backup leak alone should not expose connector credentials.

Input validationZod schemas validate bodies, params, row projections, JSON columns, and summary points.Protects service logic from malformed inputs and schema drift.
Error responsesKnown AppError types return structured errors; unknown failures return generic 500s.Reduces accidental stack/SQL leakage through HTTP.
Prompt localitySystem prompts live in src/summarizers/prompts.ts.Keeps provider instructions reviewable instead of scattering prompt text across services.

Security findings

PriorityFindingEvidenceRecommended design
S1 Deno runtime permissions are broad for API, CLI, tests, capture, and eval tasks. deno.json grants unscoped --allow-net, --allow-env, --allow-read, --allow-write, --allow-sys, and often --allow-ffi. Split dev/test/prod task profiles. Scope network hosts, readable paths, writable paths, and allowed env names. Keep FFI only where native packages require it.
S2 Summarization is a privacy boundary, not an internal call. OpenAICompatibleSummarizerService sends system prompt plus item text/media to ${baseUrl}/chat/completions. Default hosted endpoint is Gemini-compatible unless LOCAL_API=true. Expose a deployment decision: hosted provider allowed vs local-only. Document provider retention terms outside code, add per-user/source “do not send media/text to hosted LLM” policy if needed, and log provider/model selection without message content.
S3 No app-level HTTP security headers or explicit Origin/CSRF validation were observed. src/server/app.ts wires routes and the error handler only. No CSP, HSTS, X-Content-Type-Options, frame policy, referrer policy, or Origin guard is set in app code. Add security headers middleware. For POST/PATCH/DELETE, validate Origin or Referer against configured origin. SameSite=Lax and JSON preflight are helpful but should not be the only CSRF defense for cookie auth.
S4 Rate limiting is process-local and can be bypassed in multi-instance deployments. rate-limit.ts uses a module-level Map. The default key trusts x-forwarded-for or x-real-ip before falling back to global. For one VPS, document as best-effort only. Before public multi-instance deployment, use Redis/Postgres-backed shared counters and accept forwarding headers only from a trusted proxy path.
S5 Database transport security is not enforced by code. src/db/client.ts calls postgres(url) directly. README examples use local postgres://... URLs. For remote production DBs, require sslmode=require or verify-full in DATABASE_URL, or enforce TLS in client configuration. Local Docker DB can stay plaintext on loopback.
S6 Session hardening is good but not complete for a credential-holding service. Cookie name is session, TTL is 30 days, touchSession() exists in the repository layer but requireAuth currently only validates and sets userId. Rename cookie to __Host-session. Decide shorter absolute TTL and sliding last-seen behavior. Add MFA/WebAuthn/TOTP to the security roadmap before broader multi-user exposure.
S7 Operational error messages are sanitized but not key-redacted. summarizeErrorForOps() strips Drizzle params and truncates at 500 chars, but does not redact provider keys or credential-bearing URLs. Add redaction patterns for known API-key shapes and URL userinfo before persisting digest_run_feeds.errorMessage.
S8 No request body size limit was observed at the Hono app boundary. Routes call context.req.json() directly; no body-size middleware appears in buildApp(). Add a conservative JSON body limit. Most API bodies should be small; connector auth and profile updates do not need large payloads.
S9 listDigestRunFeedsForRun is a fragile helper, not a current exposed IDOR. The current route first scopes findDigestRunForUser(database, id, context.var.userId) and only calls the helper with the owned run.id. The helper itself filters only by runId. Pass userId into the helper and join/filter through digest_runs.userId so future callers cannot accidentally expose arbitrary run feed rows.
Highest-impact security fix: narrow production Deno permissions. That single change reduces the blast radius of a compromised dependency, connector bug, accidental eval import, or unexpected code-execution path more than most local code refactors.
Scalability review

The current shape supports one process and moderate usage; the next bottlenecks are connector batching, LLM payload limits, and distributed coordination.

Scalability strengths

StrengthObserved implementationWhy it scales
User pagingScheduler pages users by (createdAt, id) with default page size 100.A daily tick does not need to load all users into memory at once.
Bounded per-process user concurrencyDEFAULT_USER_CONCURRENCY = 5 in digest-job.ts.Prevents one process from starting all users at once.
Idempotent item cacheitems_feed_id_external_id_unique and upsert behavior.Overlapping windows can refresh changed items without duplicate rows.
Idempotent summaries and digestsSummary and digest repositories use onConflictDoUpdate.Retries can converge to one row per feed-period or user-period.
Partial failure modelDigest-run feed rows track connector, ingestion, and summarization stages.Operators can see which feed failed without throwing away successful feeds.
Historical digest resilienceFeed soft delete and summary feed-name snapshots preserve old digests.User history survives feed removal and upstream rename.

Scalability findings

PriorityFindingEvidenceRecommended design
P1 Ingestion repeats connector work once per feed within the same source. ingestUserFeeds() groups feeds by source and creates one connector handle, but then loops feeds and calls ingestFeed(). ingestFeed() calls connector.getNormalizedData(window.from, window.to, [feed.externalId]). Add ingestFeedsForSource(): compute one source window, call getNormalizedData(from, to, externalIds) once, validate items per feed, then upsert each feed in its own transaction or bounded batch.
P2 Summarization has no hard context, item, image, or byte cap. buildContentParts() contains a warning that 300+ group messages may overflow context, but still sends all selected messages/media in a single provider request. Introduce a summarization budget: text token/byte cap, image count cap, chunked map-reduce summarization for long threads, and explicit “truncated due to budget” metadata.
P3 Scheduler coordination is process-local. scheduleDigestJob() registers a scheduler callback in each process. runningUsers is a module-level Set and only prevents overlap inside one runtime. Before running multiple API instances or workers, guard scheduled work with Postgres advisory locks or a dedicated job table with leased rows.
P4 In-flight digest runs have no crash recovery state transition. runForUser() creates digest_runs.status = running and marks finished at the end. A process crash between those points leaves running rows. On startup or scheduler tick, mark stale running runs older than a threshold as failed/stale, then retry based on period and idempotent upserts.
P5 Local media retention is unbounded. Connector media paths use local media directories; README notes Telegram photos are downloaded for channels. No TTL, quota, or cleanup worker is present in the reviewed flow. Add lifecycle: content-addressed paths or per-run dirs, reference counting or age-based cleanup, disk quota alerts, and eventual object storage if multi-host.
P6 No timeout/cancellation boundary around external connector and summarizer calls. fetch() in the summarizer has retries for 429/503 but no AbortSignal. Connector calls are awaited without a visible timeout wrapper. Add deadline propagation from digest run to connector and summarizer calls. Record timeout as a first-class operational error.
P7 Summaries are generated sequentially per feed. assembleDigestForPeriod() loops feeds and awaits summarizeOwnedFeedPeriod() one at a time. Add small configurable summarizer concurrency after context caps are in place. Keep per-provider rate limits and per-user fairness.
P8 Digest listing is not explicitly paginated at the route/API shape. GET /digests returns listDigestViewsForUser(database, userId). Digest-run listing has a default limit in the repository; digest listing should expose pagination consistently. Add cursor or limit/offset pagination to digest and run listings before histories become large.
P9 Database pool behavior is implicit. src/db/client.ts calls postgres(url) with defaults. Set explicit max connections, idle timeout, connect timeout, and SSL mode per deployment. This matters once scheduler workers and API requests share the same database.
P10 Digest upsert is already race-safe. upsertDigestForPeriod() uses insert(...).onConflictDoUpdate(...).returning() on (userId, periodStartMs, periodEndMs). No change needed for a check-then-insert race. Keep this pattern for future period-scoped tables.
Prioritized roadmap

Do hardening before horizontal scale; do payload budgets before summarizer concurrency.

Phase 1 — production safety baseline

WorkWhy firstAcceptance criteria
Scope Deno permissions for production tasks. Reduces blast radius across every component. Production start task whitelists network hosts, env names, read paths, and write paths. Tests cover startup with the scoped profile.
Add HTTP hardening middleware. Cookie-authenticated app needs explicit browser security posture. Responses include HSTS, nosniff, frame denial, referrer policy, and a documented CSP strategy. State-changing routes reject unexpected Origin/Referer.
Add request size limits and operational error redaction. Small controls prevent cheap memory and leakage failures. Oversized JSON request returns a controlled 413. Provider/API-key patterns and credential-bearing URLs are redacted before persistence.
Define LLM privacy policy in configuration. User messages and media can leave the app. Admin/user-visible setting states hosted vs local provider. The app can refuse hosted LLM calls for protected feeds or deployments.
Enforce summarizer budgets. Prevents context overflow, unbounded cost, and giant media payloads. Long threads are chunked or truncated deterministically with recorded metadata. Tests cover text-only, media-heavy, and empty-after-filter cases.

Phase 2 — multi-user and multi-instance readiness

WorkWhy secondAcceptance criteria
Shared rate limiter.Required once more than one API process can receive traffic.Redis/Postgres counters enforce the same login/register/connector/digest limits across instances and ignore spoofed forwarding headers unless request came from a trusted proxy.
Distributed scheduler lock or leased job table.Prevents duplicate scheduled digests in multi-instance deployment.Only one worker processes a user-period at a time. Stale leases expire safely.
Crash recovery for digest runs.Operational truth must survive process death.Startup/tick reconciles stale running rows to failed/stale and can retry idempotently.
Explicit database pool and TLS configuration.A scheduler plus web traffic can exhaust implicit defaults.Pool size, connect timeout, idle timeout, and SSL mode are documented and enforced per environment.
Credential custody hardening.Telegram session strings are full-account bearer tokens.Master key moves off-box to KMS/Vault/secret manager, backup encryption uses a separate key, and MFA is available for user accounts.

Phase 3 — throughput and cost efficiency

WorkWhy laterAcceptance criteria
Batch ingestion per source.Removes repeated connector calls and upstream scans.One connector call fetches all feed external IDs for a source-period; per-feed errors remain isolated.
Connector feed lookup caching/direct lookup.Telegram dialog scans are expensive at scale.Requested feeds resolve directly or through cached mapping instead of scanning all dialogs for every run.
Bounded summarizer concurrency.Improves latency after budgets and provider rate controls exist.Configurable per-process and per-user concurrency with provider 429 backoff and fairness.
Media lifecycle and object storage.Needed for multi-host and disk-bound deployments.Media has TTL/quota cleanup, deterministic paths, and optional object-store backend.
Paginated digest history.Prevents unbounded dashboard payloads.Digest and digest-run APIs expose cursor pagination and the web UI loads incrementally.
