Current architecture, security posture, and scalability review

This document captures the repository architecture as implemented on 2026-07-19 and records the security and scalability decisions for the current Deno/Hono API, SolidStart web UI, PostgreSQL persistence layer, Telegram and Substack connectors, and OpenAI-compatible summarization pipeline.

Executive summary

Architecture is cleanly layered; production hardening is not complete.

What is strong
  • Repository functions consistently accept userId and scope reads/writes to ownership.
  • Session tokens are random 256-bit values; only SHA-256 token hashes are stored.
  • Connector credentials use envelope encryption with AES-256-GCM and owner-bound AAD.
  • Item and summary writes are idempotent through database unique constraints and upserts.
  • Service seams are simple: connectors fetch/normalize, summarizers summarize, repositories persist.
What blocks safe production scaling
  • Deno tasks grant broad network, filesystem, environment, system, and FFI permissions.
  • Rate limiting is process-local and trusts request forwarding headers without an app-level trusted-proxy policy.
  • Digest scheduling is single-process: Deno.cron plus an in-memory runningUsers set.
  • Summarization is a provider boundary with deployment-wide text and vision endpoints, deterministic text/item/image caps, and run-local vision fallback behavior.
  • Telegram/media artifacts are written locally with no TTL, quota, or object-storage lifecycle.
Important corrections
  • No current digest-run IDOR was confirmed: GET /digests/runs/:id first loads the run with findDigestRunForUser(database, id, context.var.userId), then lists feed rows for the owned run.
  • listDigestRunFeedsForRun(database, runId) is still a fragile repository helper because it is not self-scoped; pass userId if another caller is added.
  • upsertDigestForPeriod is already atomic: it uses insert(...).onConflictDoUpdate(...).returning(). There is no current check-then-insert race in digest upsert.
Security decision: Morning Post is not zero-knowledge. Scheduled digests require the backend to decrypt connector credentials without the user present. The honest target is narrower: a database or backup leak alone should not expose credentials; provider boundaries should be explicit; operational access should be minimized and auditable.
System context

Morning Post is a multi-user digest service with external connector and LLM boundaries.

The web UI calls the Deno API with cookie credentials. The API owns authentication, connector credential custody, feed subscription state, ingestion, summarization, digest assembly, and scheduled runs. PostgreSQL stores users, sessions, encrypted source credentials, feed subscriptions, item cache, summaries, digests, and digest-run audit rows.

Trust boundaries

  • Browser → API: cookie-authenticated requests. Origin guard middleware on unsafe requests validates Origin or Referer against the configured ALLOWED_ORIGINS. Security headers include HSTS, frame denial (DENY), and nosniff. Body size limited by MAX_REQUEST_BODY_BYTES.
  • API → PostgreSQL: persistence for sensitive metadata and encrypted credentials. DB_SSL_MODE configures TLS (default disable for local loopback; require or verify-full for remote). Pool size, idle timeout, and connect timeout are explicitly configured from env vars.
  • API → Telegram: Telegram session credentials are full-account bearer tokens. Server custody is unavoidable for scheduled digests.
  • API → Substack: users explicitly import substack.sid and may also provide connect.sid when their browser has it; Morning Post never accepts a Substack password or raw cookie header. Account-subscription discovery and paid/full post retrieval send only the provided session cookies to the fixed https://substack.com origin. Manual publication validation fetches public archives without cookies through the IP-pinned HTTPS boundary. The integration replays an unsupported browser session and may conflict with Substack terms; the UI discloses that risk before connection.
  • API → summarizer provider: message text, prompt text, and selected media leave the app to the configured OpenAI-compatible text and vision endpoints. ALLOW_REMOTE_SUMMARIZATION (default false) controls non-loopback providers; loopback endpoints are always allowed. Same-model requests are multimodal; distinct endpoints use vision analysis followed by text summarization. Text, item, and image-byte caps apply, with [IMAGE_OMITTED] and run-local [IMAGE_ANALYSIS_UNAVAILABLE] fallback markers.
  • API → filesystem: media are local files with feed-isolated paths, per-connector quota (MEDIA_QUOTA_BYTES), per-period cleanup, and weekly TTL sweep (MEDIA_TTL_MS).

Runtime and startup

Local startup is explicit about listener addresses, cron capability, and the browser proxy path.

The verified local run reaches the backend health endpoint at http://127.0.0.1:3000/health and renders the SolidStart frontend at http://127.0.0.1:5173. The runbook contract below records the startup failures reproduced during that verification and the fixes that now keep the two listeners reachable.

Reproduced blockers:
  1. The server tasks registered Deno.cron without enabling Deno's unstable cron capability. Startup therefore failed before the API could serve health; both server tasks now include --unstable-cron.
  2. Deno.serve defaulted its listener hostname to 0.0.0.0, while the named api permission set allowed only 127.0.0.1:3000. The process therefore failed before it could serve health; the server now passes the resolved hostname explicitly.

Corrected startup contract

Runtime Contract Verification target
Backend deno task dev:api and deno task start load .env.production.local, pass --unstable-cron, and serve with { hostname, port }. The hostname precedence is an explicit server override, then SERVER_HOSTNAME, then the default 127.0.0.1; the local default port is 3000. GET http://127.0.0.1:3000/health returns the backend health response.
Frontend npm run web:dev invokes vinxi dev --port 5173. Vinxi's outer HOST defaults to 127.0.0.1; the nested Vite server is strict on port 5173 and proxies /health to http://127.0.0.1:3000. Start the backend first because the proxy does not replace the backend health dependency. The browser renders at http://127.0.0.1:5173, and GET /health through that origin reaches the backend.
Local database cleanup deno task db:cleanup truncates every public table in the development database selected by DATABASE_URL, resets identities, and preserves the schema and migration history. The command has loopback-only network permission and rejects PostgreSQL system databases plus names ending in _test or _e2e. All development application rows are removed without allowing the destructive command to connect to a remote, test, or E2E database.
Browser E2E npm run web:e2e starts non-reusable API and web processes on ports 3100 and 5174. The API derives a database ending in _e2e from TEST_DATABASE_URL, unless E2E_DATABASE_URL supplies an explicit isolated database. Guards reject either configured development or backend test database. The E2E database is migrated and truncated before startup and truncated again in global teardown. The browser suite passes while development data is unchanged, and the isolated E2E database contains no users after teardown.
Production listener permissions SERVER_HOSTNAME must exactly match the <listener> host passed to scripts/production-start.template.sh, because that host is an explicit entry in --allow-net. Use 127.0.0.1 for a reverse-proxy-bound loopback process; use 0.0.0.0 for a directly exposed process. The process can bind only the reviewed listener and remains reachable through the intended proxy or direct endpoint.

Domain model

The persistence model is generic across connectors.

The app models connector accounts as sources and individual subscriptions as feeds. Items and summaries are cached per feed. Digests are user-facing period aggregates derived from summaries. Digest-run tables are operational audit trails for each manual or scheduled run.

For Substack, an item is an inaccessible paid post exactly when NormalizedItem.meta.audience === "only_paid" and NormalizedItem.meta.contentAccess === "preview". The orchestration layer never sends that item to the summarizer and makes no model call for it; it persists the available article metadata with contentAccess: "paid" and points: []. Accessible paid-full articles and free previews still follow normal article summarization. This representation records a title that Substack exposed; it does not mean Morning Post can read the paid body or automatically subscribe the user.

Every DigestView includes paidPosts. Digest projection partitions paid articles out of ordinary sections, preserving stable digest/article order for title projection only when the owning Substack source has showPaidPostTitles enabled. A section containing only paid articles is omitted, but a genuinely empty article section remains. Because this partition is derived at read time from immutable summaries and the source's current setting, toggling the opt-in re-renders historical digests without re-ingestion, re-summarization, or a model call. The Markdown API representation and web digest append a final title-only Paid posts section; paid articles never appear in regular sections and no preview or body text is rendered.

Current failure isolation

A connector creation failure records a failed connector-stage run feed. Per-feed ingestion and summarization failures are caught and recorded; successful feeds still contribute to a partial digest. If an unexpected error escapes those boundaries after the run row exists, runForUser() makes a best-effort transition of that exact run to failed with sanitized error text, then preserves and rethrows the original error.

Connector-specific ingestion and bounded deadlines

Telegram handles use source batching through ingestFeedsForSource(). Substack handles use ingestFeedsIndividually() with default concurrency four, one deadline per publication, and per-feed failure isolation. Feed-ingestion deadlines race connector reads and use post-await abort guards to prevent late item persistence. Substack session connection, subscribed-publication discovery, and manual publication validation all race their remote calls against the connector deadline. The connection and manual-add write paths use a commit gate that atomically cancels the deadline before starting the credential or feed database mutation, which the route then awaits normally; subscribed-publication discovery is read-only. API server tasks opt into Deno's corrected request-signal behavior, so the signal aborts for a client disconnect rather than every successful response. A response therefore cannot report a connector timeout while a connector database write continues in the background.

Security review

Primary risks are permission breadth, provider boundaries, and edge hardening.

Security controls already in place

Control Observed implementation Why it matters
Password hashing hash-wasm Argon2id with 19 MiB memory, 2 iterations, parallelism 1. Baseline resistance to offline password cracking; parameters are encoded in the hash.
Login enumeration resistance Unknown email and bad password return the same message; dummy hash path is verified. Reduces user enumeration and simple timing leaks.
Session token storage Raw 256-bit token goes only to cookie; database stores SHA-256 hash. A session table leak does not directly yield replayable cookies.
Cookie flags httpOnly: true, secure: true, sameSite: "Lax", path: "/". Protects the token from JavaScript read access and normal cross-site form posts.
Credential encryption Per-call random data key, AES-256-GCM, owner AAD, wrapped data key, KeyProvider seam. A database or backup leak alone should not expose connector credentials.
Substack credential minimization Strict schemas require substack.sid and accept connect.sid when the authenticated browser provides it. Session validation reads the user ID from userSettings[0].user_id (also accepting the documented user_settings alias), distinguishes provider/protocol failures from an expired-session 401, and revalidates before fetching subscribed publications from the fixed Substack origin. Subscription discovery joins publication records to subscription publication IDs, paginates by provider cursor, preserves provider order, and rejects repeated cursors. The web client clears local secret state after submission; API responses expose only public source or publication-feed metadata. Avoids password custody, raw-cookie ingestion, accidental credential echoing, cross-origin cookie forwarding, and rejecting valid accounts that do not receive the compatibility cookie.
Substack SSRF boundary Publication URLs require HTTPS domains without credentials, ports, or IP literals. A and AAAA answers must all be global unicast. Each request and redirect connects to the validated IP literal, then uses the original hostname for TLS SNI and certificate verification. Subscribed-publication domains are discovery candidates only; selecting one still runs the same archive validation before persistence. Closes DNS-rebinding TOCTOU while supporting arbitrary custom publication domains, including domains returned by account discovery, under unrestricted API network permission.
Ownership scoping Repositories generally take userId and include ownership filters; route handlers use requireAuth. Prevents cross-user object access at the persistence layer.
Input validation Zod schemas validate bodies, params, row projections, JSON columns, and tagged summary content. Protects service logic from malformed inputs and schema drift.
Error responses Known AppError types return structured errors; unknown failures return generic 500s. Reduces accidental stack/SQL leakage through HTTP.
Prompt locality System prompts live in src/summarizers/prompts.ts. Keeps provider instructions reviewable instead of scattering prompt text across services.

Most API bodies should be small; connector auth and profile updates do not need large payloads. Remediated. bodyLimit({ maxSize: options.maxRequestBodyBytes }) before routes in buildApp(). The current route first scopes findDigestRunForUser(database, id, context.var.userId) and only calls the helper with the owned run.id. The helper itself filters only by runId. Highest-impact security fix: narrow production Deno permissions. That single change reduces the blast radius of a compromised dependency, connector bug, or unexpected code-execution path more than most local code refactors.
Scalability review

The current shape supports one process and moderate usage; the next bottlenecks are connector batching, LLM payload limits, and distributed coordination.

Scalability strengths

Strength Observed implementation Why it scales
User paging Scheduler pages users by (createdAt, id) with default page size 100. A daily tick does not need to load all users into memory at once.
Bounded per-process user concurrency DEFAULT_USER_CONCURRENCY = 5 in digest-job.ts. Prevents one process from starting all users at once.
Idempotent item cache items_feed_id_external_id_unique and upsert behavior. Overlapping windows can refresh changed items without duplicate rows.
Idempotent summaries and digests Summary and digest repositories use onConflictDoUpdate. Retries can converge to one row per feed-period or user-period.
Partial failure model Digest-run feed rows track connector, ingestion, and summarization stages. Operators can see which feed failed without throwing away successful feeds.
Historical digest resilience Feed soft delete and summary feed-name snapshots preserve old digests. User history survives feed removal and upstream rename.

Scalability findings

Priority Finding Evidence Recommended design Status
P1 Ingestion repeats connector work once per feed within the same source. ingestUserFeeds() groups feeds by source and creates one connector handle, but then loops feeds and calls ingestFeed(). ingestFeed() calls connector.getNormalizedData(window.from, window.to, [feed.externalId]). Add ingestFeedsForSource(): compute one source window, call getNormalizedData(from, to, externalIds) once, validate items per feed, then upsert each feed in its own transaction or bounded batch. Remediated. ingestFeedsForSource() computes union window, calls getNormalizedData once per source, filters returned items per feed, validates and upserts each feed independently. Connector-level exception marks all pending feeds failed; per-feed errors isolated. Single-feed path kept. See src/services/ingestion-service.ts, src/services/orchestrator.ts. Tests in tests/services/ingestion-service.test.ts, tests/services/orchestrator.test.ts.
P2 Summarization has no hard context, item, image, or byte cap. buildContentParts() contains a warning that 300+ group messages may overflow context, but still sends all selected messages/media in a single provider request. Introduce a summarization budget: text token/byte cap, image count cap, chunked map-reduce summarization for long threads, and explicit “truncated due to budget” metadata. Remediated. Aggregate mode uses deterministic sequential chunking with maxTextBytesPerChunk (120k), maxItemsPerChunk (50), and maxImageBytes (1M), followed by one terminal text-only merge for multiple chunks. Article mode accepts one item, splits oversized text at UTF-8 code-point boundaries, sends media only with the first chunk, and concatenates chunk points without a cross-article merge. Oversize or unreadable images become [IMAGE_OMITTED]; distinct vision analysis uses strict indexed JSON; same-model 400/415/422 falls back for the current run. See src/summarizers/openai-compatible-summarizer.ts. Tests in tests/summarizer.test.ts.
P3 Scheduler coordination is process-local. scheduleDigestJob() registers a scheduler callback in each process. runningUsers is a module-level Set and only prevents overlap inside one runtime. Before running multiple API instances or workers, guard scheduled work with Postgres advisory locks or a dedicated job table with leased rows. Remediated. scheduler_leases table with primary key name supports atomic leader election. tryAcquireSchedulerLease("digest-job") runs before every tick; only the winner calls runDigestTick. Deno.cron triggers remain but database lease makes duplicates harmless. See src/scheduler/digest-job.ts, src/scheduler/scheduler.ts, src/server/main.ts. Tests in tests/scheduler/digest-job.test.ts.
P4 In-flight digest runs have no crash recovery state transition. runForUser() creates digest_runs.status = running and marks finished at the end. A process crash between those points leaves running rows. On startup or scheduler tick, mark stale running runs older than a threshold as failed/stale, then retry based on period and idempotent upserts. Remediated. runForUser() immediately finalizes unexpected in-process failures as failed without masking the original error. For process death or a database outage during finalization, recoverStaleDigestRuns() updates running rows older than DIGEST_RUN_STALE_AFTER_MS (default 15 min) to failed with redacted error text at the start of each leader tick. The partial unique index on digest_runs(userId) WHERE status='running' prevents duplicate active work. See src/services/orchestrator.ts and src/repositories/digest-run-repository.ts. Tests in tests/services/orchestrator.test.ts and tests/db/digest-run-repository.test.ts.
P5 Local media retention is unbounded. Connector media paths use local media directories; README notes Telegram photos are downloaded for channels. No TTL, quota, or cleanup worker is present in the reviewed flow. Add lifecycle: content-addressed paths or per-run dirs, reference counting or age-based cleanup, disk quota alerts, and eventual object storage if multi-host. Remediated. Feed-isolated paths (telegram_media/<feed-key>/<message-id>.jpg). Per-connector quota (MEDIA_QUOTA_BYTES, default 500MiB) evicts oldest before writes. Best-effort per-period cleanup after successful summarization. Weekly TTL sweep deletes files older than MEDIA_TTL_MS (default 7 days). Object storage deferred. See src/connectors/telegram/telegram-connector.ts, src/services/summarization-service.ts. Tests cover quota eviction, TTL cleanup, success/failure cases.
P6 No timeout/cancellation boundary around external connector and summarizer calls. fetch() in the summarizer has retries for 429/503 but no AbortSignal. Connector calls are awaited without a visible timeout wrapper. Add deadline propagation from digest run to connector and summarizer calls. Record timeout as a first-class operational error. Remediated. Optional AbortSignal added to Connector.getRawData, getNormalizedData, getMessagesFromEntity. ingestFeedsForSource creates per-source timeout from CONNECTOR_TIMEOUT_MS (default 120s). Telegram connector checks signal between iterations. Summarizer passes signal through chunk, retry, and merge with 3 retries for 429/503. See src/connectors/connector.types.ts, src/connectors/telegram/telegram-connector.ts, src/services/summarization-service.ts. Tests in tests/connector.test.ts, tests/services/summarization-service.test.ts.
P7 Summaries are generated sequentially per feed. assembleDigestForPeriod() loops feeds and awaits summarizeOwnedFeedPeriod() one at a time. Add small configurable summarizer concurrency after context caps are in place. Keep per-provider rate limits and per-user fairness. Remediated. SUMMARIZATION_CONCURRENCY (default 2) with worker-pool helper. Shared OpenAICompatibleSummarizerService handles per-request retries; no unbounded Promise.all. See src/services/digest-service.ts. Tests in tests/services/digest-service.test.ts.
P8 Digest listing is not explicitly paginated at the route/API shape. GET /digests returns listDigestViewsForUser(database, userId). Digest-run listing has a default limit in the repository; digest listing should expose pagination consistently. Add cursor or limit/offset pagination to digest and run listings before histories become large. Remediated. Cursor-based pagination on GET /digests and GET /digests/runs. Opaque base64url cursors with decoded schema validation. Default limit 20, max 100. Response shape: { data, nextCursor }. Web UI "Load more" appends without duplicates, disables while loading, resets after new run/delete/refresh. See src/server/routes/digests.ts, apps/web/src/api/client.ts, apps/web/src/app/Dashboard.tsx. Tests in tests/server/digests.test.ts, frontend unit tests.
P9 Database pool behavior is implicit. src/db/client.ts calls postgres(url) with defaults. Set explicit max connections, idle timeout, connect timeout, and SSL mode per deployment. This matters once scheduler workers and API requests share the same database. Remediated. postgres(url, { max, idle_timeout, connect_timeout, ssl }) with values from getConfig(). DB_SSL_MODE accepts disable (local default), require, verify-full; invalid mode rejected at startup. Same options in src/db/testing.ts. See src/db/client.ts, src/db/testing.ts. Config tests prove remote settings require TLS.

No change needed for a check-then-insert race. Keep this pattern for future period-scoped tables. No change needed. Atomic INSERT ... ON CONFLICT DO UPDATE already prevents a check-then-insert race. Prioritized roadmap

Do hardening before horizontal scale; do payload budgets before summarizer concurrency.

Phase 1 — production safety baseline

Work Why first Acceptance criteria Status
Scope Deno permissions for production tasks. Reduces blast radius across every component. Production start task whitelists network hosts, env names, read paths, and write paths. Tests cover startup with the scoped profile. Completed. Named permission sets; production template at scripts/production-start.template.sh.
Add HTTP hardening middleware. Cookie-authenticated app needs explicit browser security posture. Responses include HSTS, nosniff, frame denial, referrer policy, and a documented CSP strategy. State-changing routes reject unexpected Origin/Referer. Completed. secureHeaders(), origin guard, bodyLimit().
Add request size limits and operational error redaction. Small controls prevent cheap memory and leakage failures. Oversized JSON request returns a controlled 413. Provider/API-key patterns and credential-bearing URLs are redacted before persistence. Completed. MAX_REQUEST_BODY_BYTES; shared secret-redaction utility.
Define LLM privacy policy in configuration. User messages and media can leave the app. Admin/user-visible setting states hosted vs local provider. The app can refuse hosted LLM calls for protected feeds or deployments. Completed. ALLOW_REMOTE_SUMMARIZATION (default false); loopback always allowed.
Enforce summarizer budgets. Prevents context overflow, unbounded cost, and giant media payloads. Long threads are chunked or truncated deterministically with recorded metadata. Tests cover text-only, media-heavy, and empty-after-filter cases. Completed. Sequential chunking with text/image/item caps; bounded merge requests.

Phase 2 — multi-user and multi-instance readiness

Work Why second Acceptance criteria Status
Shared rate limiter. Required once more than one API process can receive traffic. Postgres counters enforce the same limits across instances and ignore spoofed forwarding headers unless request came from a trusted proxy. Completed. rate_limit_buckets table; trusted proxy via TRUSTED_PROXY_COUNT.
Distributed scheduler lock or leased job table. Prevents duplicate scheduled digests in multi-instance deployment. Only one worker processes a user-period at a time. Stale leases expire safely. Completed. scheduler_leases table with atomic tryAcquireSchedulerLease.
Crash recovery for digest runs. Operational truth must survive process death. Startup/tick reconciles stale running rows to failed and can retry idempotently. Completed. recoverStaleDigestRuns() at leader tick start. Partial unique index prevents duplicates.
Explicit database pool and TLS configuration. A scheduler plus web traffic can exhaust implicit defaults. Pool size, connect timeout, idle timeout, and SSL mode are documented and enforced per environment. Completed. DB_POOL_MAX, DB_IDLE_TIMEOUT_SECONDS, DB_CONNECT_TIMEOUT_SECONDS, DB_SSL_MODE.
Credential custody hardening. Telegram session strings are full-account bearer tokens. Master key moves off-box to KMS/Vault/secret manager, backup encryption uses a separate key, and MFA is available for user accounts. Deferred. KeyProvider seam preserved; KMS/MFA/object-storage belong to separate provider/product plans.

Phase 3 — throughput and cost efficiency

Work Why later Acceptance criteria Status
Batch ingestion per source. Removes repeated connector calls and upstream scans. One connector call fetches all feed external IDs for a source-period; per-feed errors remain isolated. Completed. ingestFeedsForSource() with union window and per-feed isolation.
Connector feed lookup caching/direct lookup. Telegram dialog scans are expensive at scale. Requested feeds resolve directly or through cached mapping instead of scanning all dialogs for every run. Not yet addressed. Depends on scheduled-run persistence integration.
Bounded summarizer concurrency. Improves latency after budgets and provider rate controls exist. Configurable per-process concurrency with provider 429 backoff and fairness. Completed. SUMMARIZATION_CONCURRENCY (default 2); worker-pool helper.
Media lifecycle. Needed for disk-bound deployments. Media has TTL/quota cleanup and deterministic paths. Completed. Feed-isolated paths, per-connector quota, best-effort per-period cleanup, weekly TTL sweep. Object storage deferred.
Paginated digest history. Prevents unbounded dashboard payloads. Digest and digest-run APIs expose cursor pagination and the web UI loads incrementally. Completed. { data, nextCursor } response shape; "Load more" in web UI.
